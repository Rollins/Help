This article explains what you need to know about System Monitoring! Website up-time, down-time, response-time, we monitor it! How healthy are our websites? How do we know the Lead Queue is processing leads? How can the TaskRunner help? This article tells you.

Pingdom

Every website or web service, available publicly, should be monitored by Pingdom. The service pings each site every 5 minutes. First, if a site Pingdom is monitoring doesn't respond, it will let us know via email. An advanced option is also provided, which checks for a specific string on a page, usually the title. If that string is missing, an email will be sent indicating the site is down. We have set most sites up with this advanced check. Pingdom also provides downtime, up-time and performance reports.

To view a list of the websites we monitor with Pingdom please log in using the credentials from the credentials wiki.


Google Analytics

We use GA for system monitoring as well. For example we monitor site traffic and have created alerts if the traffic dips a below a specified level unexpectedly. We also have implemented response-time reporting via GA. In order to get a better understand of how we utilize Google Analytics to monitor our websites I have created a Google Analytics Monitoring Audit case.

Website Health Monitoring

Health monitoring refers to a method of reporting errors encountered by users. We use a few different types of Health monitoring tools to monitor our websites and services.

Most ASP.Net websites use ASP.Net Health Monitoring. The exception is the Waltham Services which uses an ASP.Net plugin called Elmah to monitor which has added benefits. It was added to Waltham Services as an experiment. Furthermore unbranded sites using Wordpress use a plugin for this functionality.

Currently Health Monitoring errors are sent to different people for different websites. We should perform a health monitoring audit to organize the settings and alert recipients for the different monitoring tools.

Task Runner Monitoring

The Rollins Task Runner can be configured to monitor many different systems within the Rollins branded infrastructure. For example, the currently configured LeadQueueReportStuckLeadsTask and LeadQueueReportBadCodesTask monitor for leads which RCSC rejects and leads with invalid web codes, respectively. If the conditions for which they monitor are met, an email alert is sent out. Going forward, we will perform a Task Runner Audit which will implement new alerts, and organize current alerts. 

Furthermore, the TaskRunner outputs all of its debug information into the trace.txt file located in the same directory as the Windows service. When reviewing alert data, this trace file can be used to gather more information on a specific error. The service is always running on the application server (69.20.10.148) and is located in C:\Program Files\Rollins\Apps\TaskRunner\WindowsService.

Occasionally, when deploying a new version of the TaskRunner, or emptying the Lead Queue the trace.txt file can provide extra information for trouble-shooting. Also, when deploying a new version of the TaskRunner, you will need to stop the service. This is a good time, to delete the current trace.txt, because this file can get large. It isn't necessary to backup the trace.txt file, because the processes run on scheduled intervals and any existing issues will reappear after restarting, if they aren't fixed.

External Monitoring Tools

Occasionally, we come across some really helpful monitoring tools, which aren't necessarily part of our infrustructure's daily health. Here is a list and description of tools which provide monitoring.

What's my DNS?

What's my DNS is a service that pings your site from locations around the world and returns the IP address from those pings. This is very helpful when adding or updating site DNS entries. For example, if we changed the DNS of an unbranded site from one host to another, we could use this service to determine when the updated DNS entry has propagated around the world. Also, note that if the DNS entry is shown as propagated, and internally (ping from local machine) the DNS isn't resolving properly, this is most likely an internal DNS issue and should be reported to Michael Curtis in IT immediately.
