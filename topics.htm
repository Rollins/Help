The Liminal Process contains two workflows. The first is the customers calling and starting the Lead/Sold/Start sales cycle. The second occurs once daily and processes data recorded during the sales cycle.

Import AT calls into MDB - The daily process begins by importing calls related to AvidTrak sessions into the Marketing Database (MDB). To determine which calls are associated with AvidTrak, each number in the AvidTrak replacement bank is associated to a unique Marketing Campaign and VDN. Each Marketing Campaign is associated with a Marketing Program. Marketing Programs have a field indicating its inclusion in AvidTrak.  

Export calls to AvidTrak - Once calls are imported into the MDB they are subsequently exported to AvidTrak and marked as exported.

Match exported calls in AT session - AvidTrak matches the start and end times of their recorded sessions to the call connect and terminate times of the exported calls.  

Match click data in MDB - Download click data from AvidTrak paid client APIs and match it to calls stored in the MDB. At this stage it is important to match only paid clicks because only they need to be exported to Marin. These calls will now have their associated click data stored in the MDB.

ODS generate conversion file - Using the VDNs associated with leads, ODS matches lead/sold/start data to call VDNs in the MDB and generates a conversion file with revenue data. Matching is done on calls for 10 days previous to ensure enough time is given to close the lead/sold/start sales loop.

Implementation Approach

The implementation assumes that:

  • Marin is in place.
  • A "closure feed" has been established to the bid platform for web leads, sales, and starts.
  • AvidTrak Licensed Model

The goal is to use AvidTrak to improve the closure feed to include phone sales and starts. In order to tie calls to lead/sold/start information stored in ODS, call data from the call-tracking solution must be tied to a Case ID in Edge (view process overview diagram).

Tying a call to a Case ID

AvidTrack provides, via the Avidtrak Webservice API

  1. The caller's phone number (via Caller ID: caller).
  2. The number called (the solution's provided number: tracking_number).
  3. Timestamps for when the call was dialed (call_datetime) and ended (duration).
  4. Markeiting Campaign ID (requires customization by AvidTrak).
  5. Mkwid (requires customization by AvidTrak).
  6. Creative ID (adid).
  7. Bid Keyword Text (search_keyword).
  8. Match Type (match_type).

A Call Detail Record (CDR) feed from RCSC must be setup to contain the following information:

  1. The caller's phone number (via Caller ID).
  2. The number of the line dialed.
  3. Time-stamps for when the call was received and answered. Preferably in UTC.
  4. The Case ID associated with the call.

The CDR feed will be loaded to AvidTrak which will match each call to keyword, creative, and other fields tracked by AvidTrak. This data will be loaded from AvidTrak to RollinsIntegration, and ultimately back into ODS where it can be linked to case status (lead, sold, start) and the revenue file can contain phone data.

Closing the Loop

Once data is available in ODS, we must "close the loop" within Marin. All data will be contained with ODS and therefore need to be exported and uploaded in a format specified by Marin as part of an automated process.

At a minimum, the export must contain:

  • A unique identifier recognizable by Marin.
  • The call outcome (lead, sold, started).
  • Any additional data required by Marin.

Additional data should be included if at all possible:

  • Contract value (or better yet, actual revenue / lifetime value -- currently unavailable within ODS).
  • Call time.
  • Call duration.

At this point, fully automated bid optimization based on estimated profit or revenue should be possible.â€‹ Should actual revenue numbers become available in ODS then optimization can be done based on actual revenue.

Dealing with Sale and Start Latency

Aggregate data will be uploaded to Marin for the previous day and include the number of leads, sales, and starts that occurred on that day. Sale and start data will be based on the date the sale or start occurred and not the date that the lead was generated so latency is not expected to be an issue.

AvidTrak Account Structure and Reporting Implications

ODS-sourced reporting presently relies on the relationship between Marketing Campaign ID and VDN. Within ODS, a phone call is associated with a Marketing Category (which belongs to a Sub-Program, which belongs to a Program) by associating it with a Marketing Campaign ID in the RollinsIntegration.MarketingCampaigns table which maintains the campaign/category relationship.

A key risk inherent with AvidTrak (or any such solution) is that the relationship between the Marketing Campaign and its associated phone number and VDN data cannot be maintained. Therefore, to ensure the relationship between a call and it's place within Marketing's reporting structure are maintained, AvidTrak must be split into separate accounts; one per Marketing Category. This ensures that Program, Sub-Program, and Category will be maintained without requiring significant rework of the underlying structure of ODS.

The implications of creating a one-to-one relationship between AvidTrak Account and Marketing Category are:

  • Aggregate data at the Program, Sub-Program, and Category levels will be consistent for year-over-year comparisons.
  • Call-level data by Marketing Campaign will no longer be accurate.

Rosetta Stone

Rollins, Google, Marin, and AvidTrak all use different terminology and naming conventions. Use the following table to translate.

Rollins Term Google Term Bing Term Marin Term AvidTrak Term Translation Notes
Marin Keyword ID NA NA mkwid or [uniq_id] ??? A unique identifier generated by Marin.
Creative ID {creative} {adid} prcid or [creative_id] adid A unique identified generated by the advertising platform (e.g. AdWords)
Bid Keyword Text {keyword} {keyword} pkw or [keyword_text] keyword The keyword that was bid upon. This is not necessarily the keyword that was queried by the user.
Match Type {matchtype} {MatchType} pmt or [matchtype] match_type The match type of the bid that was triggered by the search query.
 

 

ODS Calls File Export

The ODS Calls File Export is a CS V formatted file, sourced from CMS and Edge, generated daily, for import into RollinsIntegration. The file can be found on the network share: \\rwinfah201\infa_shared\TgtFiles\Prod\AvidTrak and contains the following fields:

Column Name Description
CALLING_PTY The ten digit caller ID phone number of the calling party.
FIRSTVDN The first VDN the call was routed through. This ultimately indicates the number that was dialed.
CALLID The call identifier assigned by Edge. This is also used as the primary key in the Calls table
SEGSTART The date and time the call was placed.
SEGSTOP The date and time the call ended.
CASEID The Edge Case ID tied to the lead.

Call Detail Record Export

The Call Detail Record Export is a CSV formatted file, sourced from RollinsIntegration for import into AvidTrak, that contains the following fields:

Column Name Description
CallOriginatingNumber The ten digit caller ID phone number of the calling party.
CallDialedNumber The ten digit number of the dialed line.
CallConnectUtc The date and time (in UTC) the call was connected.
CallTerminateUtc The date and time (in UTC) the call was terminated.

Creative ID A value generated by the ad platform that uniquely identifies an ad unit.
Keyword The bid-upon keyword.
Match Type The match type of the bid that was triggered by the search query.
weblead2 Conv

webstart Rev The revenue, as indicated by the contract value, generated on the specified date for the specified keyword ID, creative ID, keyword, and match type that was generated from web leads.
phonelead Conv The number of cases associated with the call. Multiple leads for the same call. Please see Linking a Call to a Case ID => Special and Unusual Cases below.

  • QA.ORKIN.COM â€“ This server is owned and administered by the Marketing Technology team. The server is used by the team to review completed code, to ensure quality and confirm the code is free of defects.
    Only a Marketing Technology team administrator can push code to this server
  • WWW.ORKIN.COM â€“ This server is the production environment, and contains the code that comprises the public-facing content of Orkin.com. The web-based software on this server is the Orkin.com website. Code deployed to this server is considered complete and released.
    Only a Marketing Technology team administrator can deploy the website to this server.
